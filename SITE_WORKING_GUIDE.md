## How URLs Are Resolved (Technical)

### Local Development
1. Frontend loads, sees empty `API_URL` in config.js
2. Detects development mode
3. Uses Vite proxy (configured in vite.config.js line 55)
4. Routes `/api/*` → `http://localhost:3001`

### Production
1. Frontend loads, reads `API_URL` from config.js (auto-generated by GitHub Actions)
2. `API_URL` = `https://d2u93283nn45h2.cloudfront.net`
3. All API calls go through CloudFront
4. CloudFront routes `/api/*` → API Gateway
5. API Gateway → Lambda (with Cognito auth enforced)

### Configuration Files
| File | Purpose | How It Works |
|------|---------|-------------|
| `webapp/frontend/public/config.js` | Local dev template | Empty API_URL triggers proxy |
| `webapp/frontend/dist/config.js` | Production generated | GitHub Actions creates this |
| `vite.config.js` | Proxy configuration | Routes local /api/* requests |
| `webapp/frontend/src/services/api.js` | URL resolution | Smart detection of environment |

## Troubleshooting

### Issue: Local Frontend Shows API Errors
**Check 1:** Is your backend running?
```bash
curl http://localhost:3001/api/health
```

**Check 2:** Is dev server running?
```bash
# In webapp/frontend directory
npm run dev
```

**Check 3:** Check browser console for network errors
- Open DevTools (F12)
- Look for fetch/XHR errors
- Verify URL being called

### Issue: Production Returns 401 Errors
This is **expected** - API requires Cognito authentication. To test:
1. Log in via /login endpoint
2. Cognito will provide ID token
3. Token is sent in Authorization header automatically

### Issue: Production 403 Forbidden
You're authenticated but don't have permission. Check:
- Cognito user pool settings
- API Lambda authorization rules
- CloudFront origin access control

### Issue: Changes Not Reflecting in Production
```bash
# Push to main to trigger deployment
git push origin main

# Check deployment status
gh run list --workflow deploy-code.yml --limit 1

# Check CloudFront invalidation
gh run view <run-number> --json conclusion
```
