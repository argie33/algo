# Stock Analytics Platform — Architecture & Setup

## The One True Server

**There is ONE API server:** `webapp/lambda/index.js`

It runs as plain Express locally and deploys to AWS Lambda in production. All other root-level server files (`local-server.js`, `api-server.js`, `server.js`) are **deprecated** — do not use them.

---

## Local Development Setup

### Prerequisites
- Node.js 20+
- PostgreSQL 14+ running locally (or configure `DB_HOST` for remote)
- `.env.local` file with database credentials

### Start the Stack

```bash
# Terminal 1: Start the API server (port 3001)
node webapp/lambda/index.js

# Terminal 2: Start the unified frontend (port 5174)
cd webapp/frontend
npm run dev

# Open browser
http://localhost:5174
```

The frontend proxy at `/api/*` automatically routes to `http://localhost:3001/api/*`.

## Critical Database Table Names

Use these exact names in queries (not `price_history_daily` or `technicals_daily`):

```
price_daily          (not price_history_daily)
price_weekly         (not price_history_weekly)  
price_monthly        (not price_history_monthly)
technical_data_daily (not technicals_daily)
buy_sell_daily       (not "signals" table)
buy_sell_weekly
buy_sell_monthly
```

## API Response Format

All endpoints return ONE of these formats:

### Success (single object)
```json
{
  "success": true,
  "data": { /* object */ },
  "timestamp": "2026-04-25T03:30:00.000Z"
}
```

### Success (paginated list)
```json
{
  "success": true,
  "items": [ /* array */ ],
  "pagination": {
    "limit": 50,
    "offset": 0,
    "total": 4969,
    "page": 1,
    "totalPages": 100,
    "hasNext": true,
    "hasPrev": false
  },
  "timestamp": "2026-04-25T03:30:00.000Z"
}
```

### Error
```json
{
  "success": false,
  "error": "Descriptive error message",
  "timestamp": "2026-04-25T03:30:00.000Z"
}
```

## AWS Deployment

### Architecture
```
CloudFront (S3 + API Gateway proxy)
    ↓
API Gateway (regional) → /api/* routed to Lambda
    ↓
Lambda (webapp/lambda/index.js via serverless-http)
    ↓
RDS PostgreSQL (VPC, private subnet)
```

### Deploy
```bash
cd webapp/lambda
serverless deploy
```

Or via SAM:
```bash
sam build --template template-webapp-lambda.yml
sam deploy
```

## Data Loading Rules (CRITICAL)

### The Process: Local First, Then AWS
1. **Load locally** using Python loader scripts (39 official loaders only)
2. **Verify completeness** locally before pushing
3. **Deploy to AWS** - CI/CD builds Docker images automatically

### Official Loaders (39 Total)
See `DATA_LOADING.md` and `LOADER_STATUS.md` for complete list with phases.

### ✅ DO
- Only create loaders that are in the 39-item official list
- Load ALL data locally before deploying
- Run loaders in proper dependency order (Phase 1 → Phase 6)
- Skip records with missing critical data
- Only insert real Buy/Sell signals (never insert 'None' signals)
- Document new loaders in DATA_LOADING.md BEFORE implementing

### ❌ DON'T
- Create extra weird loaders (`loadindustryranking.py`, etc.)
- Use fake populate scripts (`populate-all-signals.js`, `generate-signals-efficient.js`)
- Insert default values like COALESCE to 0 or 'None'
- Patch data in API routes instead of using loaders
- Add test/debug scripts in root directory
- Push partial/incomplete data to AWS
- Mix loader logic into route files

**Why?** Loaders got out of control in the past with 53+ Python files, 66+ Dockerfiles, and fake populate scripts inserting bad default data. Data loading MUST be centralized and disciplined.

## Troubleshooting

### No data showing on frontend
1. Check `/api/health` → should return `{ success: true, ... }`
2. Check `/api/diagnostics` → shows which tables are populated
3. Check browser DevTools Network tab → are API calls returning 200?
4. Check API URLs in `webapp/frontend/src/services/api.js` — endpoints must start with `/api/`

### Database connection failed
1. Verify `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD` in `.env.local`
2. Test connection: `psql -h localhost -U stocks -d stocks -c "SELECT 1"`
3. Check RDS security group if using AWS

### Port 3001 already in use
```bash
# Kill the process
lsof -i :3001 | grep LISTEN | awk '{print $2}' | xargs kill -9
# Or specify different port
PORT=3002 node webapp/lambda/index.js
```
