These are **NOT code issues** - they're configuration issues in AWS. You must verify/set these:

#### 1. **RDS Proxy Configuration**
**Check:** AWS Lambda environment variable `DB_HOST`

**Must contain:** The word "proxy"

**Example (correct):**
- `algo-rds-proxy.c2q3x4a1b2c3.us-east-1.rds.amazonaws.com` ✓

**Example (WRONG):**
- `algo-database.c2q3x4a1b2c3.us-east-1.rds.amazonaws.com` ✗ (direct RDS)

**How to fix:**
```bash
# In AWS Lambda console:
# Go to: Functions → algo-api-dev → Configuration → Environment variables
# Update DB_HOST to include "proxy"

# Or via AWS CLI:
aws lambda update-function-configuration \
  --function-name algo-api-dev \
  --environment Variables={DB_HOST=algo-rds-proxy.c2q3x4a1b2c3.us-east-1.rds.amazonaws.com,...}
```

#### 3. **Cognito Configuration** (if using authentication)

**Source:** All Cognito values are generated by Terraform and passed through deployment workflows. NEVER hardcoded.

**Terraform Outputs (terraform/outputs.tf):**
- `cognito_user_pool_id` - User pool ID for frontend auth
- `cognito_user_pool_client_id` - App client ID for frontend auth
- `cognito_domain_url` - OAuth domain URL (e.g., https://algo-dev.auth.us-east-1.amazoncognito.com)

**How it flows to frontend:**
1. `terraform apply` generates outputs with actual Cognito values
2. `deploy-all-infrastructure.yml` reads outputs and passes to `build-prod.js`
3. `build-prod.js` generates `dist/config.js` with values injected
4. Frontend loads `config.js` to configure Cognito authentication

**Validation (CRITICAL):**
Both `deploy-all-infrastructure.yml` and `deploy-code.yml` NOW FAIL EXPLICITLY if any Cognito output is empty/null. This prevents silent auth failures from using wrong or missing Cognito domains.

**If deployment fails with "COGNITO_* from Terraform is empty":**
1. **Check Terraform outputs:** `cd terraform && terraform output | grep cognito`
2. **Verify Cognito user pool exists:** `aws cognito-idp list-user-pools --max-results 10 --region us-east-1`
3. **Verify Cognito domain exists:** `aws cognito-idp describe-user-pool-domain --domain algo-dev --region us-east-1`
4. **Check Terraform state:** `terraform state show module.cognito` to verify resources are created

**If Cognito resources don't exist:**
Run full `deploy-all-infrastructure.yml` workflow (not just `deploy-code.yml`) to create them.

**Development mode (for testing):**
```bash
# To bypass Cognito in dev/staging:
DEV_BYPASS_AUTH=true
```

⚠️ **WARNING:** `DEV_BYPASS_AUTH=true` must NOT be set in production!

#### 3b. **CloudFront Cache Invalidation (Automatic)**

**What it does:**
Both `deploy-all-infrastructure.yml` and `deploy-code.yml` automatically invalidate CloudFront cache after syncing frontend assets to S3. This ensures users receive fresh content immediately after deployment.

**Flow:**
1. Frontend assets are synced to S3 with proper cache headers
   - `config.js` gets `Cache-Control: no-cache` (never cache)
   - HTML files get `Cache-Control: no-cache` (revalidate always)
   - Other assets get `Cache-Control: max-age=31536000, immutable` (cache forever)
2. CloudFront cache is explicitly invalidated with `--paths "/*"`
3. Browser cache-busting query strings are added to script tags in HTML (e.g., `config.js?v=1234567890`)

**Why both approaches are needed:**
- Query strings alone don't clear CloudFront's cache (CloudFront caches the base file)
- Cache headers tell browsers/CDN how long to cache, but don't force immediate invalidation
- `create-invalidation` immediately clears CloudFront's cache for all paths

**If invalidation fails:**
The deployment will FAIL EXPLICITLY with error message: `ERROR: CloudFront distribution ID not found` or `ERROR: Failed to create CloudFront invalidation`

This prevents silent cache failures where users would receive stale `config.js` with outdated API URLs.

## COMMON ISSUES & SOLUTIONS

### Issue: "Cannot read properties of null" errors in console

**Root Cause:** Component receiving null data from API

**Solution:**
1. Check API response in F12 Network tab
2. Verify `statusCode: 200` but `data: null`?
3. Run config validator: `python lambda/api/config_validator.py`
4. Check data pipeline execution

---

### Issue: "API service temporarily unavailable" message

**Root Cause:** Circuit breaker opened (12+ consecutive failures)

**Solution:**
1. Check Lambda logs for actual errors
2. Fix the root cause (DB connection, schema error, etc.)
3. Wait 15 seconds for circuit to recover
4. Or restart Lambda function

---

### Issue: CORS errors in F12 console

**Root Cause:** FRONTEND_URL not set or CloudFront domain not in Secrets Manager

**Solution:**
1. Verify FRONTEND_URL in Lambda environment
2. Or store CloudFront domain in Secrets Manager: `algo/cloudfront-domain`
3. Redeploy Lambda
4. Clear browser cache

---

### Issue: 5xx errors on specific endpoints only

**Root Cause:** Database table not populated or missing schema

**Solution:**
1. Check data pipeline execution status
2. Verify table exists: `SELECT * FROM market_exposure_daily LIMIT 1`
3. If empty, run pipeline manually or wait for next scheduled run
4. Check `CRITICAL_ISSUES_FOUND.md` Issue #3 (Empty Tables)
